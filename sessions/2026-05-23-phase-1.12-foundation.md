## What shipped

### Step 1 — Infrastructure
- **`app.py`** — `VIEW_MODES = ('fancy','plain','reader')` + `DEFAULT_VIEW_MODE = 'fancy'`; `resolve_view_mode()` (signed-in DB pref → `view_mode` cookie → default, returns `(mode, is_set)`); `inject_globals()` now injects `view_mode`, `view_mode_set`, `view_decorated`, `display_surface`. `_USER_PREF_WRITES['poem_view_mode']` validator widened to the new vocabulary. Removed the per-route `server_view_mode` resolution from `poem_permalink`.
- **`base.html`** — `<body class="view-{{ view_mode }}{% if view_decorated %} view-decorated{% endif %}">`. Removed the 1.29 localStorage `data-view-mode` propagation script.
- **`poem.html`** — removed `data-view-mode`/`data-server-mode`/`data-signed-in`/`data-short-code` (dead) from the article, the per-poem Plain/Pasture toggle control, and the client-side resolution `<script>`.
- **Collections/bio/queue** — removed hardcoded `data-view-mode="pasture"` from `my_pasture.html`, `saved_horses.html`, `user_profile.html`, `poem_queue.html`; they now follow the site mode.
- **`style.css`** — migrated every `[data-view-mode='pasture']` selector to `body.view-fancy` (faithful, preserves the current Fancy look); reduced-motion now suppresses *animation within* the active mode (motion decoupled from mode — the old JS `reduced-motion → plain` downgrade is gone); deleted the dead `.poem-pasture-view`/`.pasture-chip`/`.pasture-line`/`.poem-permalink-box` block and the orphaned `.poem-view-controls`/`.poem-view-toggle` rules.

### Step 2 — Picker
- **`app.py`** — `POST /set-view-mode` route: no-JS form post that writes the `view_mode` cookie (1-year, SameSite=Lax) and, for signed-in users, mirrors to `poem_view_mode`; redirects back via a same-site-only `next`. `DISPLAY_SURFACES` set + `is_display_surface()`.
- **`base.html`** — collapsed control in the nav (a native `<details>` — works with no JS) showing `⊙ <Mode> ▾`; first-run strip below the nav (`[Fancy][Plain][Reader]` + dismiss), shown only when `not view_mode_set and display_surface`.
- **`profile_edit.html`** — "Display mode" section mirroring the picker for signed-in users.
- **`style.css`** — picker styling (`.mode-menu`, `.mode-firstrun`, `.mode-prefs-form`, `.mode-opt`) + mobile-nav rules. Flagged provisional.

### Step 3 — Plain skin
- **`app.py` / `base.html`** — `view_decorated` flag + `body.view-decorated` class (Fancy + Plain; Reader is undecorated).
- **`style.css`** — grass backdrop + cream note-box panels now **shared** by both decorated modes via `body.view-decorated`; coat `--bg/--fg` split out as shared, with the scattered tilt kept Fancy-only; new `body.view-plain .poem-horse` functional colour-pill chips (coat colours + famous shimmer, **no body parts, no tilt**, bumped padding); famous shimmer + its reduced-motion suppression broadened to `view-decorated`.

## Decisions (with provenance)

- **confirmed** — Session scope = steps 1+2+3 as one PR (Clover chose "full mechanical foundation" over step-1-only). *Why:* step 1 alone removes the only mode switcher; bundling keeps a coherent, mergeable state.
- **ok** (Clover deferred to judgment, implicitly via "continue") — Pulled the *mechanical* `[data-view-mode='pasture']` → `body.view-fancy` selector rename into step 1 (the spec lists it under step 4). It's a no-op rename preserving today's look, so it doesn't jump the prototype gate (which is about *new* Fancy styling). Keeps master from showing undecorated Fancy between commits.
- **ok** (judgment) — Resolution lives in the existing `inject_globals()` context processor; reads the already-loaded `g.current_user` pref via `get_preferences` (one indexed read on template renders only).
- **ok** (judgment) — Picker is **no-JS-first**: form posts to `/set-view-mode` → cookie + redirect; collapsed control is a native `<details>`. Picking a mode is a full reload (server-emitted class). Matches the spec's "works with JS disabled" + Web 1.0 ethos. No JS progressive-enhancement layer added (deferrable).
- **ok** (judgment, **flagged for Clover**) — `view-decorated` shared class: Fancy and Plain share the entire grass-backdrop + note-box treatment; they diverge only on chip decoration. This realizes the spec's own observation that "a workhorse page in Fancy ≈ how Plain treats every page," and sets up step 4 to split Fancy off into "text floats on grass."
- **noted** — Bounded the Plain field scope: Plain gets grass on the four display/collection containers (not a new site-wide body background). The site-wide "field is the surface" treatment + the Fancy bio-poem transparent-to-grass fix stay in the **gated step 4**, as the spec assigns them.
- **noted** — Reader currently falls back to the base/Plain-ish look (cream `.poem-view`, dotted-underline chips, no field). Its real typographic treatment is the gated **step 5**.

## Testing holds — Clover, please verify live

1. **Mode resolution / first paint (no JS):** with JS disabled, each of the three modes renders correctly server-side (no flash, no dependence on script).
2. **Picker — anon:** fresh visitor (no `view_mode` cookie) on a display surface (`/p/<code>`, `/featured`, `/browse`, a `/u/<slug>`, `/me/pasture` etc.) sees the first-run strip with Fancy highlighted. Picking Plain/Reader and dismissing both set the cookie and hide the strip on reload. Collapsed nav `⊙ <Mode> ▾` opens and switches on every page.
3. **Picker — signed-in:** picking a mode persists to the account (`poem_view_mode`) and survives a new session / different device; the "Display mode" section on `/me/profile` reflects and changes it.
4. **All three modes across surfaces:** permalink, Featured, Browse, a public profile (bio poem), My Pasture, Saved Horses, Saved Poems, and the admin queue (`/admin/poem-queue`). Fancy = today's pasture look; Plain = colour pills on grass, no legs/tail; Reader = plain text baseline (interim).
4b. **Workhorse pages** (editor, admin, account forms, count, legal) — chrome carries the mode, editor chips stay sober (no body parts).
5. **Reduced-motion:** with `prefers-reduced-motion: reduce`, Fancy keeps its decoration but the famous-shimmer and chip tilt stop — and the mode does **not** switch.
6. **Stored-value migration:** any account with an old `plain`/`pasture` pref resets to the Fancy default (acceptable per pre-beta posture).
