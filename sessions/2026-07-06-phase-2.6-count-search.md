## What shipped

A Clover user-suggestion, specced ‚Üí built ‚Üí deployed ‚Üí live-verified in one session:
search the dictionary by **how many horses share a name**, inside the existing Wildcard
search box (no new UI mode). Three forms:

- **Pure count:** `>8` (shared by more than 8), `>=8`, `=2` / bare `8` (exactly).
- **Combined:** `*love*>8` ‚Äî the wildcard `*love*` search restricted to names shared by
  more than 8 horses. `star*>=10`, etc.
- Wildcard-only + count (`*>8`) collapses to pure count.

### Why it was cheap
The number is `len(registrations)` ‚Äî already computed and returned as the `count` field
on every search result. **No data-model change, no new search toggle.** All new code is
in `poetry.py`: `_parse_count_query` (peels the trailing count expression) + `_count_search`
(the branch), routed from the top of `search_dictionary`. `/poetry/search` endpoint, rate
limit, and JSON shape untouched ‚Äî the existing chip renderer + pager needed nothing.

### The load-bearing fact (checked, not assumed)
Clover's instinct was to overload the search box because "numbers and `<>=` are disallowed
in horse names." Verified against the **full dictionary (2,123,231 normalized names)**:
**zero** names contain a digit, **zero** contain `<`/`>`/`=`; the entire character set is
`a`‚Äì`z` + space. So a trailing count-expression can *never* collide with a name ‚Äî the parse
is unambiguous. This is an **invariant**, noted at the parse site and in the spec: if
dictionary normalization ever admits digits/symbols, the grammar breaks.

### Combined search is *faster* than the wildcard it's built on
Applying the O(1) count predicate **before** the regex shrinks the set the regex scans.
Measured on the full dict: `*love*>8` ‚âà 294 ms vs plain `*love*` ‚âà 1280 ms (regex runs on
~22k names instead of 2.1M). Nice inversion of the usual "extra filter = slower."

### Safeguards
- **Floor:** reject any predicate that includes count==1 (`=1`, `>=1`, `>0`) ‚Äî that's 84%
  of the dictionary (1,790,534 count-1 names). Friendly error, not a dump.
- **Cap ordering:** almost every count query trips `SEARCH_HARD_CAP = 2000` (you don't drop
  under 2000 total until ~`>=24`). Pure-count results sort **count desc, then name** and
  slice ‚Äî a capped result keeps the *most-shared* names, so the cap reads as intentional
  ("top 2000 most-shared"). Exact `=N` is all-tied ‚Üí alphabetical ‚Üí labelled `(sample)`.
  Combined results sort by the existing name-relevance rule (count is a filter, not the
  sort key).
- **3-char minimum** applies to the name part *only*, after peeling the count (so `>8` and
  `*>8` become pure count instead of failing the length check).

### Discoverability (agreed non-negotiable)
The syntax is undiscoverable, so it ships *with* teaching: the "? How it works" modal gained
a bullet with three **clickable** worked examples (`>8`, `=2`, `*star*>20`) that close the
modal, switch to Wildcard, and run the search (`runExample()` + `.suggest-search` styling).

## Decisions made
- **Reuse the search box, no new mode** ‚Äî confirmed (Clover's call, backed by the charset
  scan). Cheaper and no grammar collision.
- **v1 operators = `=`, `>`, `>=` only.** `<`/`<=` deferred ‚Äî degenerate on this data (every
  "less than" includes the count-1 bucket ‚Üí ~2M matches). As built they return a targeted
  "not supported yet" error rather than a silent empty name search.
- **`[opus ¬∑ high]` actual vs `[sonnet ¬∑ medium]` spec estimate** ‚Äî ran on Opus (Clover
  switched mid-session); the surface was as small as estimated, so either would've done.

## Two spec deviations (reconciled in the spec, now FINAL)
1. **sort-and-slice, not `heapq.nlargest`** ‚Äî equivalent; worst case (`=2`, ~332k tuples)
   sorts well inside the rate-limit budget, so bounded-memory heap buys nothing.
2. **`<`/`<=` ‚Üí friendly error**, not "fall through to name search" ‚Äî strictly better UX.
