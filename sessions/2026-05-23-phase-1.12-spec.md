---

## Goal

Replace the binary `plain` / `pasture` display system with **three named modes** the user picks site-wide, resolved server-side, and applied as a single skin across the whole site.

The current system (as-built): a per-container `data-view-mode` attribute flips between `plain` (base) and `pasture` (decoration layered on top); resolution runs client-side in JS (server pref → localStorage → `prefers-reduced-motion` → pasture default); a per-poem toggle button switches the two. Phase 1.29 added a `base.html` script that propagates the localStorage mode to non-permalink containers.

1.12 throws out the binary, moves resolution to the server, and makes mode a property of the *whole page*, not just the poem.

---

## The three modes

Labels are **descriptive**, not thematic (Clover's call — "pasture" was shorthand for the fancy one because it has horses in a pasture, but as a *label* the mode is "Fancy" because it gets the fancy styles).

| Mode | What it is | Chips | Surface |
|---|---|---|---|
| **Fancy** | All the fancy styles. The immersive mode. | Full body-part chips, coats, rotation, famous-shimmer | The field *is* the surface; text/UI floats on grass, separated by a contrasting outline; non-horse text de-emphasized; UI ornate |
| **Plain** | Functional CSS only — the workhorse / accessible mode. | Colored chips + shimmer, **no body parts** | Field is a decorative *backdrop*; each content area is a count-page-style pinned note-box; bumped text + touch targets |
| **Reader** | Text-only, minimal design. | Typographic, no decoration | Off-white page, print-stylesheet aesthetic on screen (the 1.11 font toolbox); woodprint-style button borders; **no field** |

Naming bonus: moving the mode name to **Fancy** frees **"Pasture"** to mean only the *place / collection* (the `/pasture` page, "My Pasture"). The long-standing double meaning is retired.

---

## Scope: full site-wide skin, bounded

**Decision: the mode is a full site-wide skin, but the functional guts of workhorse pages stay utilitarian.**

- **Chrome follows the mode everywhere** — nav, footer, and page background take on the active mode on *every* page, for visual coherence. Driven by a single mode class on `<body>` (or `<html>`).
- **Display / reading surfaces fully follow the mode** — poem permalink (`/p/<short>`), Featured, Browse, Random, public profiles (`/u/<slug>`), the collection pages (My Pasture, Saved Horses, Saved Poems), and profile bio poems.
- **Workhorse guts stay utilitarian regardless of mode** — the "Wrangle a Poem" editor canvas, admin pages (`/admin/*`, legacy `/submissions`), account-management forms (`/me/profile`, `/setup-account`, sign-in), the count tool (`/count`), and legal pages. Their *chrome* still skins; their *interactive content* stays sober. Editor chips never get body parts (honors the standing "UI vs display separation" + "no body-part whimsy on editor chips" commitments).

Note this is internally consistent: a workhorse page in **Fancy** mode = grass-backdrop chrome + a sober content box on top, which is essentially how **Plain** mode treats every page. So "workhorse guts stay utilitarian" is not a special case to hand-build per page — it's the Plain treatment applied to those content regions regardless of site mode.

**Why bounded and not "everything goes ornate":** maximal immersion (ornate editor/admin in Fancy) contradicts the editor-chip commitment and balloons the build/maintenance surface. Rejected.

---

## Resolution: server-emitted class

**Decision: cookie (anon) + DB pref (logged-in), server emits the mode class. No JS required to apply the skin.** This closes the migration-audit open design question ("view-mode fallback chain runs client-side → should be server-authoritative for accessibility").

- **Logged-in:** `users.preferences_json.poem_view_mode` (already exists; allow-list validator extended to accept `reader`). Server reads it and emits the class.
- **Anonymous:** a single functional cookie `view_mode` (`fancy` | `plain` | `reader`). No consent-banner concern for a strictly-functional preference cookie; the site's privacy posture is already relaxed on this. The existing `syncLocalToAccount` path mirrors it on first login.
- **No cookie / no pref → default = Fancy.** A first-time visitor lands in the full fancy experience.
- The server emits the class on first paint, so there is **no flash-of-wrong-mode** and **no JS dependency** — works with JS disabled.

**Motion is decoupled from mode** (change from current behavior — flagged because it touches the accessibility commitment, accepted by Clover deferring to judgment): today `prefers-reduced-motion` *downgrades pasture → plain*. It will no longer switch modes. Instead a `@media (prefers-reduced-motion: reduce)` block suppresses *animation* (transforms, future walk-cycle) inside whatever mode is active. A reduced-motion user keeps Fancy's coats / body-parts / grass — they just don't get movement. This also means the server never needs a client-only signal to resolve the mode. (Moot for 1.12 since Fancy ships static, but the rule is set now so animation lands cleanly later.)

## The mode picker

**One control, two states, anchored at the top of the page. The absence of the `view_mode` cookie IS the "fresh user" signal — no separate tracking.**

- **First-run (no cookie, display surfaces only):** a prominent **non-modal strip directly under the nav** showing `[ Fancy ] [ Plain ] [ Reader ]` with **Fancy highlighted**. Makes it obvious to a newcomer that if the full fancy isn't for them, calmer options exist. Shown only on display surfaces (where the skin is most dramatic) — workhorse pages just show the collapsed control.
- **Any interaction** — picking a mode *or* dismissing — writes the cookie (dismiss writes the default `fancy`). From then on:
- **Collapsed (persistent, every page):** a compact control **in the nav** — `⊙ Fancy ▾` (or similar) — that re-expands the picker on click. Also mirrored in account preferences for logged-in users.

**Placement rationale:** the collapsed control lives in the nav because that's the natural, view-source-able home for a site-wide preference, and it avoids a fixed floating widget (fits the Web 1.0 ethos better than a corner overlay). The first-run strip sits right below the nav — i.e. right under where the control permanently lives — and "tucks up" into the nav on interaction. **Tradeoff named:** to change mode later, an anon reader scrolls back to the top. Fine for a set-and-forget preference. If real use shows constant mid-read flipping, add a fixed pill *then* — not built speculatively.

## Renderer rearchitecture

- **Mode class moves from per-container to site-level.** Today `data-view-mode` sits on individual poem containers. It becomes one class on `<body>` / `<html>`, server-emitted. CSS keys off that.
- **The 1.29 `render_chip` / `render_poem` macros stay the single render source.** They already emit coat/rev/famous hooks that are inert until CSS acts on them — that design carries straight over; the macros don't need to know the mode.
- **Per-container hardcoded modes are removed.** 1.29 hardcoded `data-view-mode="pasture"` on collection pages and bio poems so they'd always render decorated. Under a true site-wide mode they **follow the chosen mode** instead — in Plain mode the pasture page shows plain chips; in Reader mode it shows text-only names. (How a horse-*collection* page reads in Reader mode is a prototype-review question — see below.)
- **Dead CSS removed.** The `.poem-pasture-view` / `.pasture-chip` / `.pasture-line` block (`style.css` ~1312–1356) is unreferenced by any template (confirmed this session — the only match was an unrelated element `id="pasture-chips"`). Delete it as part of the rearchitecture.
- **CSS organization:** three mode-namespaced skins selected by the body-level class. Plain is effectively the base/simplest; Fancy layers decoration (migrate the current `[data-view-mode='pasture']` rules to the Fancy skin); Reader is a distinct typographic treatment.

**Stored-value migration:** values become `fancy` | `plain` | `reader`. Old stored `plain` / `pasture` values reset on this change — fine per pre-beta operating posture (Clover is the only user; break freely). Allow-list validator (`_USER_PREF_WRITES`, `app.py`) updated to the new vocabulary.

## Prototype-first (build options; commit only after Clover reviews)

Per the ROADMAP commitment, text styling in Fancy/Reader is decided by reviewing rendered options, not pre-specified. Carve an explicit **prototype review gate** into the build before these are committed:

1. **Fancy text styling** — the contrasting-outline color separating floating text/UI from the grass (color TBD at prototype), and the non-horse-text de-emphasis treatment.
2. **Reader text styling** — print aesthetic on screen using the 1.11 font toolbox (Smokum / Abril Fatface / IM Fell English / Playfair Display SC) + woodprint-style button borders.
3. **Fancy mobile** — floating text on a narrow viewport. The genuinely hard layout problem; prototype before committing.
4. **Horse-collection pages in Reader mode** — what a grid/list of horse *names* (My Pasture, Saved Horses) looks like stripped to text. May warrant a per-surface exception; decide at prototype.
