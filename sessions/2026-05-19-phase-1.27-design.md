## Decisions made

**1.27 architecture: draft-centric stable model** · `[Clover proposed, Claude approved]`
The stable is not a general per-user pool. It belongs to a specific poem/draft. Every user always has at least one draft ("New Poem" stub). "Add to stable" always targets a specific draft.

**Option C: per-poem stables with picker** · `[Clover proposed, Claude steelmanned, held position → Clover overrode]`
Claude recommended Option B (one active draft, simpler). Clover's overriding use case: you're browsing your pasture and spot a horse perfect for a poem you shelved a week ago — Option B requires navigating away and losing the horse. Option C (picker) is the right call. Claude agrees post-override.

**Picker rule: 1 draft → add directly; 2+ drafts → show picker** · `[Clover proposed, Claude approved]`
Picker is triggered whenever any named/saved draft exists alongside the "New Poem" stub. "New Poem" is always listed in the picker. Picker lives inline in the chip menu — content replaces in place, Esc goes back.

**Re-save updates in place** · `[Clover proposed, Claude approved]`
Save Draft on a loaded draft overwrites it. Draft identity tracked via `draft_id` in JS state seeded at load time.

**Anonymous users: one current poem in localStorage** · `[Claude proposed, Clover confirmed]`
Anonymous draft object stores lines + stable + metadata combined (not just stable as today). No multi-draft for anon. On login, syncs to a "New Poem" draft. The `saveStableLocal()` no-op approach is replaced.

**Drafts page publish flow** · `[Clover proposed, Claude confirmed]`
Publish from drafts routes through `/submit/poem` per existing trust rules (admin = instant, everyone else = queue). Same path as editor publish. No special-casing needed in 1.27.

**Drafts page features** · `[Clover proposed, Claude confirmed]`
- List with title + date
- Expandable preview (poem lines or stable)
- Delete with confirm gate
- "Resume editing" → auto-saves current editor state first, then switches (no lost work, no prompt)
- Publish action (routes per trust model)

**Trust score system design** · `[Clover proposed, Claude confirmed]`
Integer per user, starts at 0. +1 per admin-approved poem with no tag edits; -1 per poem where admin edits tags. Admin manually overrides score from user page. Per-action thresholds stored in DB (`auto_post_threshold = N`). Current behavior (admin bypass, all else queue) is threshold-∞ / threshold-0 special case — no change until 1.13.1 ships. Logged to ROADMAP as 1.13.1, depends on 1.13.

**`queue_handler.py` untouched in 1.27** · `[Claude proposed, Clover confirmed]`
New draft storage goes in SQLite (`drafts` table, extended). Ephemeral 1-hour TTL in queue_handler.py is a separate concern, fixed when that stage is directly addressed.
