### Cross-posting platform research

**Strategic reframe:** goal is now funnel-to-site (social posts as hooks, poet.horse as destination) rather than the old content-delivery model (Tumblr as terminal). This simplifies connector design — one permalink per post, no per-horse PQ links — and lowers link-ban risk across all platforms.

**Tumblr ban clarification:** the ban was most likely triggered by per-horse PedigreeQuery links (many different external URLs per post), not poet.horse links. The new connector architecture is a meaningfully different profile.

**Platform findings:**

| Platform | Priority | Notes |
|---|---|---|
| Bluesky | 1st | Open AT Protocol, app-password auth, no bot hostility; link cards don't count against 300-char limit; literary community growing |
| Mastodon | 2nd | Open REST API, bot-friendly, 500-char limit, CW field maps to site tags; friction = instance selection |
| Threads | 3rd (image-only) | Only viable at low volume or image-only; consider if Facebook auth is added to Clerk; Meta bureaucracy + link suppression otherwise |
| X | Skip | API costs + bot hostility + link suppression; not worth building |

**Image posts:** images convert better than plain text on every platform, sidestep character limits, and avoid link-suppression penalties. Image-card export promoted to early Phase 2 as a cross-posting prerequisite.

### Playwright dependency — two-for-one

Server-side Playwright (preferred image-card export technique) also enables Cloudflare-aware PQ scraping. Same dep, same VPS setup. This makes the Playwright path clearly correct: install once, use for both.

**VPS coordination:** Playwright needs system-level shared libraries requiring root (`playwright install-deps chromium`). One-time setup. Coordinate with Jon (rupture server admin) before planning the build.

---

### PQ lazy-cache scrape design

On poem publish, enqueue any uncached horses for PQ lookup. Background worker drains the queue at very low rate (one request per N minutes, queue-wide — not per-horse, so publish bursts don't create request bursts). Cache is permanent-until-manually-invalidated (PQ data for a given horse is static). Data to capture: coat color, breed, country, birth year, sire/dam. Feeds real coat colors on chips; pedigree available for future per-horse pages. Do not build before Playwright is confirmed on the VPS.
